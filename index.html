一、gcc与g++区别

gcc集成了各种不同语言的编译器，代表GNU Compiler Collection，g++是gcc的c++编译器。编译代码是根据代码的后缀名来判断调用编辑器。例如后缀是.c，会调用c编译器还有linker去链接c的库；如果后缀是.cpp，会调用g++编译器，调用的库也是c++版本的。

二、gcc编译

1、一次性编译

直接使用gcc加文件名编译，如果想自定义输出文件名再加-o参数

2、分步编译

  • 编译（compile）：将源文件编译成目标文件需要用 -c 选项，想自定义输出目标文件名需要 -o 选项
  • 链接 （link）：gcc后面紧跟目标文件的名字，就可以将目标文件链接成可执行文件，想自定义可执行文件的名字需要 -o 选项

3、整个编译流程

gcc 编译器在编译一个c语言程序时经历一下4步

  • 将c语言程序预处理，生成 .i 文件
  • 预处理后的 .i 文件编译成汇编文件 .s 文件
  • 将汇编语言文件经过汇编，生成 .o 文件
  • 将各个模块的 .o 文件链接起来生成一个可执行程序文件

gcc 编译选项

  • -c : 编译 + 汇编，生成的是 .o 文件
  • -S：编译，生成的是 .s 文件
  • -E：预处理，生成的 .i 文件
  • -I ：指定包含的头文件目录（大写i）
  • -L：手动添加链接库路径（大写L）
  • -l：添加链接库名字（小写L）
  • -g：生成能被gdb调试器所用的调试信息
  • -save-temps：将整个编译过程生成的中间文件都存下来

gcc -l：手动添加链接库

  • 链接器把多个目标文件链接成一个单独的可执行文件。在链接过程中，它需要把符号（全局变量，函数）用对应的数据内存地址（变量地址、函数地址）替代，以完成程序中多个模块的外部引用。
  • 对链接器而言，链接库不过是一个具有许多目标文件的集合，放在一个文件中方便处理，比如C标准库函数
  • 标准库的大部分函数通常放在文件 lib.a(.a代表archive，获取的意思)或者放在用于共享的动态链接文件 lib.so(shared object，共享对象的意思)。这些链接库一般位于 /lib 或者 /usr/lib，或者gcc默认搜索的其他目录
  • 链接程序时，gcc默认链接 lib.a 和 lib.so，其他的库就需要手动添加

链接其他目录中的库

  • 链接库作为一般目标文件，为gcc指定链接库的完整路径与文件名

  • 使用 -L选项，为gcc增加另一个搜索链接库的目录，可使用多个 -L，也可在路径内使用冒号分割列表

  • 把链接库目录加到环境变量 LIBRARY_PATH 中

三、动态库与静态库

静态库在链接期间被组合到程序中，动态库在程序运行阶段被加载进内存

1、静态库生成

linux下静态库的命名规则为 libxxx.a，xxx表示的是库的名字。gcc生成静态库的链接分两步

  • gcc把源文件编译为目标文件
  • ar命令将.o文件打包成静态库链接

2、静态库使用

使用静态链接库时，除了需要库文件本身，还需要头文件；库文件包含了函数定义的代码（编译后），头文件则包含了函数声明的代码。为了使用静态库，我们创建一个文件夹math，作为新项目的基础目录。

规范的项目目录如下：lib文件夹一般存放库文件，include文件夹一般存放头文件，src文件夹一般存放源文件，bin文件夹一般存放可执行文件。为了规范，将上文的 libtest.a 放在lib下，test.h 放在include下，在src下创建一个main.c文件

编译main.c时，我们需要使用-I（大写的i）选项指明头文件的包含路径，使用-L选项指明静态库的包含路径，使用-l（小写的L）选项指明静态库的名字

3、动态库生成与使用

gcc生成动态库的参数有两个：-shared选项，输入文件可以是源文件、汇编文件和目标文件；-fPIC选项，作用于编译阶段，告诉编译器产生位置无关代码，产生的代码没有绝对地址只有相对地址，就可以被加载器加载到内存中任意位置。

必须要确保程序运行时可以找到动态库，可以将动态库放到标准目录下，例如/usr/lib；或者设置一个合适的环境变量，例如LIBRARY_PATH。

编译器的工作过程.md

发表于 2020-04-01 | | 阅读次数:
字数统计: 1.7k 字 | 阅读时长 ≈ 5 分钟

源码要运行，必须先转成二进制码，这是编译器的任务。复杂的项目，编译过程还需分成三步。

一、配置（configure）

编译器在工作之前，需要知道当前的系统环境，比如标准库在哪里、软件的安装位置在哪里、需要安装哪些组件等等。这是因为不同计算机的系统环境不一样，通过指定编译参数，编译器就可以灵活适应环境，编译出各种环境都能运行的机器码。这个确定编译参数的步骤，就叫做”配置”（configure）。

这些配置信息保存在一个配置文件之中，约定俗成是一个叫做configure的脚本文件。通常它是由autoconf工具生成的。编译器通过运行这个脚本，获知编译参数。

二、确定标准库和头文件的位置

源码肯定会用到标准库函数（standard library）和头文件（header）。它们可以存放在系统的任意目录中，编译器实际上没办法自动检测它们的位置，只有通过配置文件才能知道。

编译的第二步，就是从配置文件中知道标准库和头文件的位置。一般来说，配置文件会给出一个清单，列出几个具体的目录。等到编译时，编译器就按顺序到这几个目录中，寻找目标。

三、确定依赖关系

对于大型项目来说，源码文件之间往往存在依赖关系，编译器需要确定编译的先后顺序。假定A文件依赖于B文件，编译器应该保证做到下面两点。

  • 只有在B文件编译完成后，才开始编译A文件
  • 当B文件发生变化时，A文件会被重新编译

编译顺序保存在一个叫做makefile的文件中，里面列出哪个文件先编译，哪个文件后编译。而makefile文件由configure脚本运行生成，这就是为什么编译时configure必须首先运行的原因。

四、头文件的预编译

不同的源码文件，可能引用同一个头文件（比如stdio.h）。编译的时候，头文件也必须一起编译。为了节省时间，编译器会在编译源码之前，先编译头文件。这保证了头文件只需编译一次，不必每次用到的时候，都重新编译了。

不过，并不是头文件的所有内容，都会被预编译。用来声明宏的#define命令，就不会被预编译

五、预处理

预编译完成后，编译器就开始替换掉源码中bash的头文件和宏。编译器在这一步还会移除注释。

六、编译

把源码转为汇编码，转换后的汇编码文件称为目标文件（object file）

七、链接

目标文件还不能运行，必须进一步转成可执行文件。编译器的下一步工作，就是把外部函数代码（后缀通常为.lib和.a文件）添加到可执行文件中，叫链接（linking）。

make命令的作用，就是从头文件预编译开始到做完这一步。

八、安装

上一步的连接是在内存中进行的，即编译器在内存中生成了可执行文件。下一步，必须将可执行文件保存到用户事先指定的安装目录。

表面上，这一步很简单，就是将可执行文件（连带相关的数据文件）拷贝过去就行了。但是实际上，这一步还必须完成创建目录、保存文件、设置权限等步骤。这整个的保存过程就称为”安装”（Installation）。

九、操作系统连接

可执行文件安装后，必须以某种方式通知操作系统，让其知道可以使用这个程序了。比如，我们安装了一个文本阅读程序，往往希望双击txt文件，该程序就会自动运行。

这就要求在操作系统中，登记这个程序的元数据：文件名、文件描述、关联后缀名等等。Linux系统中，这些信息通常保存在/usr/share/applications目录下的.desktop文件中。另外，在Windows操作系统中，还需要在Start启动菜单中，建立一个快捷方式。

这些事情就叫做”操作系统连接”。make install命令，就用来完成”安装”和”操作系统连接”这两步。

十、生成安装包

写到这里，源码编译的整个过程就基本完成了。但是只有很少一部分用户，愿意耐着性子，从头到尾做一遍这个过程。事实上，如果你只有源码可以交给用户，他们会认定你是一个不友好的家伙。大部分用户要的是一个二进制的可执行程序，立刻就能运行。这就要求开发者，将上一步生成的可执行文件，做成可以分发的安装包。

所以，编译器还必须有生成安装包的功能。通常是将可执行文件（连带相关的数据文件），以某种目录结构，保存成压缩文件包，交给用户。

十一、动态链接

正常情况下，到这一步，程序已经可以运行了。至于运行期间（runtime）发生的事情，与编译器一概无关。但是，开发者可以在编译阶段选择可执行文件连接外部函数库的方式，到底是静态连接（编译时连接），还是动态连接（运行时连接）。所以，最后还要提一下，什么叫做动态连接。

前面已经说过，静态连接就是把外部函数库，拷贝到可执行文件中。这样做的好处是，适用范围比较广，不用担心用户机器缺少某个库文件；缺点是安装包会比较大，而且多个应用程序之间，无法共享库文件。动态连接的做法正好相反，外部函数库不进入安装包，只在运行时动态引用。好处是安装包会比较小，多个应用程序可以共享库文件；缺点是用户必须事先安装好库文件，而且版本和安装位置都必须符合要求，否则就不能正常运行。

现实中，大部分软件采用动态连接，共享库文件。这种动态共享的库文件，Linux平台是后缀名为.so的文件，Windows平台是.dll文件，Mac平台是.dylib文件。

看完 Thomas 大神的视频Photoshop选区、通道、蒙版关系与用法全解析，觉得还是图文的形式比视频更好理解和记忆。开局一张图，箭头表示二者之间可以实现转换。

选区与通道

选区存储为通道

  • 右侧通道面板右下方四个按键里第二个：将选区存储为通道
  • 快捷键：Alt + S + V （存储选区）

通道转化为选区：蚂蚁线显示的是不透明度50%以上的选区（白色全选，黑色全不选）

  • 按住 Ctrl，点击该通道
  • 快捷键：Alt + S + O （载入选区）
  • 右侧通道面板下方四个按键里第一个：将通道作为选区载入
选区与蒙版

选区转化为蒙版

  • 右侧图层面板右下方六个按键里第二个：添加图层蒙版
  • 右侧图层面板右下方六个按键里第三个：添加调整图层，自动生成图层蒙版

选区替换原来的蒙版

  • 选区转化为通道

  • 将原来蒙版选中，点编辑->填充->黑色

  • 通道转化为选区

  • 将白色蒙版选中，点编辑->填充->白色

通道与蒙版

通道与蒙版之间不能转化，必须依赖于选区作中间桥梁

蒙版与蒙版

蒙版复制

  • 从一个图层复制到另一个图层：按住 ALT 键，然后拖动蒙版到另一个图层

蒙版移动

  • 从一个图层移动到另一个图层：直接拖动到另一个图层

蒙版反向

  • 选中蒙版，按 ctrl + i

无输出，不涉入——2019年通鉴

写年终总结主要有两个原因：一、如果对2019年的365个日夜进行聚类，当会有很多相似度极高的日子，能记住的本就不多。若日后检视岁月的时候，能有一些图、文、视频可据，也未尝不是一件幸事与快事。二、岁月如歌，唱着唱着就跑了调。梳理这一年的得与失，希望能将跑偏的调子再拉回来。这份人物只有自己的“资治通鉴”，就记录那些与众不同的日子、让人振袖而起的反思以及个人的进退损益。

贯穿本年度的一个核心思考是：无输出，不涉入（No Output, No Engagement）。意为一件事倘不以输出为结果，就不要去做。猛地一看可能有些矫枉过正，然正如太祖在《毛选》中说的矫枉必须过正，不过正则不足以矫枉。

居正
《春秋公羊传》有言

君子大居正

意为君子以居正道为大，其余都是细枝末节。这一年我认识到就目下而言，所谓的正道就是工作。如果工作上无进益，那么其他方面发展的再好，本质上都是在营造“我也并没有在浪费时间”的幻觉，都是对现实的逃避。在上一家公司的这一年，获益不太多，想离开但又深知自己太菜，在走与留之间摇摆了许久。中间迷茫与博士师兄聊天，师兄一席话点醒了我：如果你觉得现在是个坑，那就不管怎样都要跳出来；如果犹豫，那就别嘴上抱怨。十月开始准备了两个月，经历了很多磕磕绊绊，在元旦前终于换了一家。

孙圈圈在《请停止无效努力》一书中表达过这样的一个观点

学习无外乎三个方面：知识，能力，价值观。

大多数知识如果不转化为能力，则是毫无用处的。沟通知识与能力的桥梁，是实践。以前觉得什么有意思就找过来学一学，c++，计算机系统、python等学了一堆，后来发现如果没有应用场景，很快就会遗忘地干干净净。它们都是知识，因为没有实践（缺乏应用场景），我并没有成功地将其转化为能力。后续在工作中，当遵循按需学习的原则，需要什么就学什么，将应用作为反馈去修正学习的路径。学完后将整个学习的过程串一下，写些技术博客作为输出。

游艺
《论语》有言

依于仁，游于艺

意为道德标准以仁为依据，并以六艺陶冶身心。2019年培养了两项技艺：拍照与做饭。

2019年10月大西北回来后，决定开始学做菜。三个月时间内成绩可喜，从最开始啥也不会，到现在已经能做出一桌招待朋友的菜。向家里的大厨们讨来做菜的知识，然后不断地开火试错，盐放的不对，火候有误等坑一个个地蹚过来，也能做出像样的饭菜。

无论是拍照还是做饭，都按照“无输出，不涉入”的原则来的，照片和饭菜就是输出。新的一年，要继续将这两大爱好的边界推到尽可能远。
